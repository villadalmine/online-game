Online Galaxy War — Arquitectura

Un PoC 100% self-hosted y API-first: cómo está construido y cómo fluye el tráfico, de punta a punta.

PoC · self-hosted 100% en casa · sin nube API-first · FastAPI k3s · Kubernetes arm64 · bare-metal Cilium · Gateway API builds in-cluster

Esto no corre en ningún proveedor cloud. Todo el juego vive en un clúster k3s casero sobre placas ARM (bare-metal), detrás de la infra de red propia. La misma imagen de contenedor corre idéntica en la laptop y en el clúster; la configuración entra por variables de entorno (12-factor) y el contenido del juego es data-as-code en YAML. La premisa de diseño: toda la lógica detrás de /api/v1 — web, CLI y futuros clientes son consumidores delgados.

Flujo de tráfico internet → pod

El borde es un HAProxy propio que hace SNI passthrough: no termina TLS, sólo mira el ServerName del handshake y reenvía el byte-stream cifrado al backend = la VIP del Cilium Gateway en la LAN. El TLS se termina recién en el Gateway, con el certificado que emite cert-manager (Let's Encrypt, reto DNS-01).

componente flujo de datos ━ anotación de protocolo
  1. El cliente abre https://<dominio>; el handshake TLS lleva el host en el SNI.
  2. HAProxy (edge propio) hace passthrough: no descifra, reenvía el TCP cifrado a la VIP del Cilium Gateway en la LAN.
  3. El Gateway (Gateway API sobre Cilium) termina el TLS con el certificado que mantiene cert-manager.
  4. Un HTTPRoute matchea por hostname y enruta al Service.
  5. El Service (ClusterIP) balancea a los pods de galaxy-api (FastAPI/uvicorn), escalables por HPA.

Pipeline de build & deploy git → registry → clúster

1 · push

Commit a main

SemVer (make release): mueve el CHANGELOG, fija appVersion y el tag de imagen, taggea git.

2 · build

Kaniko / Argo

Build in-cluster en un nodo arm64, sin Docker daemon. El Dockerfile multi-stage corre la suite de tests: si algo falla, no hay imagen.

3 · registry

Registry interno

La imagen se publica en el registry del propio clúster (nada sale a internet).

4 · deploy

Helm --atomic

initContainer migrate (Alembic) + smoke; helm test; si el rollout no queda sano, rollback solo.

Defensa en profundidad del deploy: el gate de tests vive en el build (no se produce imagen rota), el smoke frena el rollout antes de exponerlo, y --atomic revierte si algo sale mal. Los secretos van por Secret de k8s; en prod, el arranque aborta si un secreto quedó en su valor por defecto.

Por qué escala (siendo casero) turn-based asíncrono

stateless

API horizontal

Los pods no guardan estado → réplicas detrás del Service, HPA por carga.

lazy state

Sin cron por usuario

Energía y producción se calculan desde *_updated_at al leer; el costo no crece con usuarios inactivos.

async

Por turnos

Sin requisitos de baja latencia → muchísimo más barato de escalar que tiempo real.

degradación

Redis opcional

Si Redis no está, cache/locks degradan a no-op: el juego sigue corriendo.

Hardware bare-metal en casa

Clúster k3s multi-nodo sobre placas ARM de bajo consumo (Raspberry Pi + Turing RK1 + Super6c) y un nodo x86 para tareas puntuales. Almacenamiento persistente en disco local del clúster (PVC). Cero dependencia de nube.

Por seguridad, esta página describe la arquitectura y los protocolos, pero omite el direccionamiento exacto (IPs privadas LAN, hostnames de nodos, endpoints internos).
