Online Galaxy War — Arquitectura

Un PoC 100% self-hosted y API-first: cómo está construido y cómo fluye el tráfico, de punta a punta.

PoC · self-hosted 100% en casa · sin nube API-first · FastAPI k3s · Kubernetes vCluster arm64 · bare-metal Cilium · Gateway API

Esto no corre en ningún proveedor cloud. Todo el juego vive en un clúster k3s casero sobre placas ARM (bare-metal), y se desplegará (planeado) en su propio vCluster para aislarlo y poder crear más objetos sin tocar el clúster anfitrión. La misma imagen de contenedor corre idéntica en la laptop y en el clúster; la configuración entra por variables de entorno (12-factor) y el contenido del juego es data-as-code en YAML. La premisa: toda la lógica detrás de /api/v1 — web, CLI y futuros clientes son consumidores delgados.

Flujo de tráfico internet → pod

El borde es un HAProxy propio que hace SNI passthrough: no termina TLS, sólo mira el ServerName del handshake y reenvía el byte-stream cifrado al backend = la VIP del Cilium Gateway en la LAN. El TLS se termina recién en el Gateway, con el certificado que emite cert-manager (Let's Encrypt, reto DNS-01). El Gateway enruta al Service del juego, que vive dentro de su vCluster.

    Pipeline de build & deploy git → registry → vCluster

    1 · push

    Commit a main

    SemVer (make release): mueve el CHANGELOG, fija appVersion y el tag de imagen, taggea git.

    2 · build

    Kaniko / Argo

    Build in-cluster en un nodo arm64, sin Docker daemon. El Dockerfile multi-stage corre la suite de tests: si algo falla, no hay imagen.

    3 · registry

    Registry interno

    La imagen se publica en el registry del propio clúster (nada sale a internet).

    4 · deploy

    Helm → vCluster

    helm upgrade --atomic dentro del vCluster del juego; initContainer migrate+smoke, helm test y rollback solo si algo falla.

    Defensa en profundidad del deploy: el gate de tests vive en el build (no se produce imagen rota), el smoke frena el rollout antes de exponerlo, y --atomic revierte si algo sale mal. Los secretos van por Secret de k8s; en prod, el arranque aborta si un secreto quedó en su valor por defecto.

    Por qué escala (siendo casero) turn-based

    stateless

    API horizontal

    Los pods no guardan estado → réplicas detrás del Service, HPA por carga.

    lazy state

    Sin cron por usuario

    Energía y producción se calculan desde *_updated_at al leer; el costo no crece con usuarios inactivos.

    async

    Por turnos

    Sin requisitos de baja latencia → muchísimo más barato de escalar que tiempo real.

    vCluster

    Aislamiento

    El juego corre en su propio vCluster: API propia, RBAC y CRDs sin tocar el clúster anfitrión.

    Hardware bare-metal en casa

    Clúster k3s multi-nodo sobre placas ARM de bajo consumo (Raspberry Pi + Turing RK1 + Super6c) y un nodo x86 con GPU Tesla P4 para la IA local. Almacenamiento persistente en disco del clúster (PVC). Cero dependencia de nube.
